Connection flow is based on 5 tuple: protocol, source IP (initiator IP), source port, destination IP (responder IP), destination port. A five tuple must exact match may it become the same flow for relayed traffic (they can match reverse, so the source IP/source port is is usually here called the 'initiator' of the connection). In the flow you record the last timestamp used.
For UDP: if a flow is not found, you create one. The last timestamp is now the last time you relayed a UDP packet. If older than 30 seconds kill the flow (UDP idle timeout)
For TCP: usually a firewall has these states (for a flow)
- NEW / SYN sent by one IP. Within a short time a SYN must be received by the other.  Timeouts are short still
- SYN_RECV, Packet seen: SYN+ACK. ACK completing the handshake. Timeouts are short still
- SYN_RECV + ACK -> ESTABLISHED after ACK acknowledging SYN/ACK. Timeout is now 15 minutes
- ESTABLISHED + FIN -> FIN_WAIT (one side sends FIN, you must track who sent FIN). Like `fin_from_initiator, fin_from_responder`. 
- FIN_WAIT + opposite FIN -> TIME_WAIT. Why? TIME_WAIT exists because the endpoint that sends the final ACK cannot know whether the peer received it. The state is therefore a safety window that absorbs late or duplicate packets from the previous connection instance.
